 1. Глоссарий
 * Клиент (Пользователь) — физическое лицо, осуществляющее поиск, подбор и бронирование отеля через интерфейс сервиса.
 * Сервис (Платформа) — веб-сайт или мобильное приложение туроператора, предоставляющее интерфейс для работы с турпродуктами.
 * Поставщик (Отель/Агрегатор) — внешняя система (отель напрямую или консолидатор вроде Ostrovok, Академсервис и др.), предоставляющая данные о доступности номеров и тарифах по API.
 * Ваучер — электронный документ, подтверждающий право Клиента на проживание в отеле на условиях оплаченного тарифа.
 * Тайм-аут бронирования (Hold time) — фиксированное время, на которое система Поставщика блокирует номер для проведения оплаты Клиентом.
 2. Бизнес-цели
 * Увеличение конверсии в продажи (СR): Предоставить клиентам возможность бронировать отели в режиме «одного окна» без перехода на сторонние площадки.
 * Рост среднего чека (Cross-sell): Реализовать допродажу проживания к уже купленным авиа/жд билетам или формирование динамического пакетного тура.
 * Снижение операционных затрат: Автоматизировать полный цикл бронирования, отмены и получения ваучеров без привлечения менеджеров поддержки (Zero-touch процесс).
 3. Описание бизнес-процесса (Целевой процесс To-Be)
 1. Поиск: Клиент вводит параметры (город/отель, даты, кол-во гостей). Сервис отправляет параллельные запросы к API Поставщиков, агрегирует, фильтрует результаты и выдает единый список.
 2. Выбор номера и тарифа: Клиент выбирает тип номера и условия (с завтраком, бесплатная отмена и т.д.).
 3. Ввод данных: Клиент заполняет ФИО гостей и контактную информацию.
 4. Предбронирование: Сервис отправляет запрос POST /booking/hold в систему Поставщика для временной блокировки номера (запускается таймер на 15 минут).
 5. Оплата: Клиент перенаправляется на платежный шлюз.
 6. Подтверждение: После успешного списания средств Сервис отправляет запрос POST /booking/confirm. Поставщик возвращает ID брони отеля.
 7. Финализация: Сервис меняет статус заказа на «Оплачено», генерирует ваучер и отправляет его на Email Клиента.
 4. User Story & Критерии приемки (Acceptance Criteria)
> Как Клиент туроператора,
> Я хочу выбрать и забронировать отель с мгновенным подтверждением и оплатой картой,
> Чтобы зафиксировать за собой номер по выбранной цене и сразу получить документ для заселения.
> 
Критерии приемки:
 * Система должна холдировать (блокировать) выбранный номер на стороне Поставщика сразу после перехода Клиента на форму оплаты.
 * Таймер на оплату составляет ровно 15 минут. На экране отображается обратный отсчет. Если оплата не совершена, бронь автоматически аннулируется (вызывается POST /booking/release).
 * Ваучер должен быть сгенерирован в формате PDF и отправлен на email в течение 2 минут после успешного ответа от платежного шлюза.
 5. Ограничения (Constraints)
 * Технические: Время ожидания (Timeout) ответа от API Поставщиков при поиске не должно превышать 6 секунд. Поставщики, не ответившие за этот срок, исключаются из выдачи, чтобы не заставлять Клиента ждать.
 * Бизнес-ограничения: Запрещено бронирование номеров, если до даты заселения осталось менее 3 часов (риск овербукинга на стороне отеля).
 * Безопасность (Compliance): Хранение и передача персональных данных гостей должны строго соответствовать ФЗ-152 «О персональных данных». Данные банковских карт на стороне Сервиса не хранятся (соответствие стандарту PCI DSS).
 6. Критерии готовности к разработке (Definition of Ready — DoR)
Задача может быть взята командой разработки в спринт, если:
 * Спроектированы и утверждены UI/UX-макеты интерфейса (экраны поиска, карточки отеля, формы ввода данных и страницы успеха).
 * Готова и согласована техническая спецификация API (Swagger/OpenAPI) интеграции со всеми целевыми Поставщиками.
 * Архитектор подтвердил техническую реализуемость предложенной схемы интеграции.
